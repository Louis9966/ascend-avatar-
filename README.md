# ascend-avatar PRD — 实时对话 2D 数字人产品需求文档

> **版本**：v1.1
> **更新日期**：2026-06-23
> **产品定位**：基于昇腾 910B 的实时对话 2D 数字人系统（单机验证版），已扩展支持“上传视频 + 文本生成 MP4”工作流
> **目标用户**：基础研究组内部验证与后续产线化评估
> **交付形态**：可在单机 Web 页面中实时对话，也可上传自定义视频生成唇形同步短视频的 2D 数字人 Demo

---

## 1. 产品定位与目标

### 1.1 产品定义

**ascend-avatar** 是一个部署在昇腾 910B 单卡上的**实时对话 2D 数字人**系统。用户输入问题后，系统通过大模型生成回复文本，并实时合成语音与唇形同步的 2D 数字人视频流，实现"像打视频电话一样"的对话体验。

### 1.2 核心体验目标

| 指标 | 目标值 | 说明 |
|------|--------|------|
| **端到端首帧延迟** | ≤ 1.5 秒 | 用户输入 → 数字人开口说话的第一帧画面 |
| **对话流式延迟** | 文本逐字/逐句输出时，音频与视频同步跟随 | 不允许等 LLM 全部生成完才开始 TTS |
| **唇形同步准确率** | 肉眼对齐 | 嘴型开合与音频内容无显著错位 |
| **支持语种** | 中文、英文、中国方言 | 优先级：中文 > 英文 > 方言 |
| **并发能力** | 单机单并发 | 当前版本只要求 1 路对话 |
| **形象数量** | 1 个默认形象 + 用户上传视频 | 默认形象用于实时对话；上传视频用于 MP4 生成 |
| **视频生成** | 上传 5–30s 正面人脸视频 + 文本 → 输出 512×512 MP4 | 使用 PaddleSpeech TTS + MuseTalk THG，25fps、H.264+AAC |

### 1.3 明确排除项

- **实时对话仍是核心场景**：视频生成为辅助工作流，不替代实时对话体验。
- **不包含 3D 数字人**：当前版本仅支持 2D 唇形同步。
- **不包含声音克隆训练流程**：声音克隆作为预留能力，当前版本只要求内置若干音色，克隆功能为后续扩展。
- **不包含 K8s/多卡集群**：当前版本只要求单机单卡（`/dev/davinci7`）容器内可运行。

## 2. 用户故事与使用场景

### 2.1 用户故事

> **作为** 一位研究者，
> **我希望** 在浏览器中打开一个页面，输入中文问题，看到 2D 数字人实时用语音回答，
> **以便** 快速验证昇腾 910B 上实时对话数字人的可行性与效果。

### 2.2 使用场景

1. **内部技术验证**：输入问题，观察数字人回答的延迟、唇形同步质量、语音自然度。
2. **中英文切换对话**：输入英文问题，数字人切换为英文回答（含英文语音和英文唇形）。
3. **方言探索**：输入方言文本，验证 TTS 对方言的支持能力（当前版本为演示性质）。

## 3. 功能需求

### 3.1 Must-Have（核心功能，必须实现）

| 编号 | 功能 | 需求描述 |
|------|------|---------|
| FR-01 | 文本输入 | 用户可在前端页面输入文本问题（支持中文、英文） |
| FR-02 | LLM 流式回复 | 后端调用外部 LLM API（OpenAI 格式，流式），实时获取回复文本 |
| FR-03 | 流式文本分句 | LLM 输出的文本按语义子句（标点切分）实时送入下游 TTS |
| FR-04 | TTS 语音合成 | 将文本子句实时合成为语音（PCM/WAV 流），支持中文、英文 |
| FR-05 | THG 唇形同步 | 根据语音流实时驱动 2D 数字人形象，生成唇形同步视频帧 |
| FR-06 | 视频流输出 | 将视频帧实时推送到前端，用户看到数字人"正在说话"的画面 |
| FR-07 | 默认形象 | 系统内置 1 个 2D 数字人形象（正面人像照片），开箱即用 |
| FR-08 | 内置音色 | 系统内置至少 2 个中文音色（男/女或不同风格），开箱即用 |
| FR-09 | 中英切换 | 根据输入语言自动切换 TTS 语言模型，或允许用户手动选择 |
| FR-10 | Gradio 前端 | 提供可交互的 Gradio Demo 页面，包含：输入框、音色选择、视频输出区域 |
| FR-21 | 视频生成工作流 | 用户可上传正面人脸视频并输入文本，后端生成唇形同步 MP4 |

### 3.2 Should-Have（重要功能，尽量实现）

| 编号 | 功能 | 需求描述 |
|------|------|---------|
| FR-11 | 首帧预热 | 系统启动后预加载模型，避免首次对话的冷启动延迟 |
| FR-12 | 流式音频缓冲 | 小窗口音频缓冲（look-ahead），覆盖 TTS 流式 chunk 的边界拼接 |
| FR-13 | 对话历史 | 前端维护多轮对话上下文，LLM 能感知前文 |
| FR-14 | 性能监控 | 前端显示实时延迟指标（TTS 延迟、THG 延迟、端到端延迟） |
| FR-15 | 方言支持 | TTS 支持至少 1 种中国方言（如粤语、四川话） |

### 3.3 Could-Have（扩展功能，当前版本不做硬性要求）

| 编号 | 功能 | 需求描述 |
|------|------|---------|
| FR-16 | 声音克隆 | 用户上传 3~10 秒参考音频，即可克隆音色 |
| FR-17 | 多形象切换 | 用户可在多个预置形象中切换 |
| FR-18 | 形象定制 | 支持上传自定义人像照片作为数字人形象 |
| FR-19 | 情感控制 | 根据文本情感调整数字人表情（微笑、严肃等） |
| FR-20 | 3D 数字人接口 | 预留 3D 数字人渲染接口（viseme/blendshape 中间表示） |

## 4. 非功能需求

### 4.1 性能需求

| 编号 | 指标 | 目标 |
|------|------|------|
| NFR-01 | 首帧延迟 | ≤ 1.5 秒（LLM 首 token → 首帧视频） |
| NFR-02 | 对话总延迟 | 中句（~20 字）端到端 < 5 秒 |
| NFR-03 | 视频帧率 | 输出视频 ≥ 25 fps，无卡顿 |
| NFR-04 | NPU 显存 | 单并发总显存占用 ≤ 16 GB（910B-32 富余 ~16 GB 用于缓冲） |
| NFR-05 | 内存 | 容器内内存占用 ≤ 32 GB |

### 4.2 可用性需求

| 编号 | 需求 |
|------|------|
| NFR-06 | 系统启动后，前端页面可立即访问 |
| NFR-07 | 单次对话失败（如 LLM 超时）不影响系统整体可用性，可重新开始对话 |
| NFR-08 | 前端页面在主流浏览器（Chrome、Edge、Firefox）中可正常访问 |

### 4.3 可维护性需求

| 编号 | 需求 |
|------|------|
| NFR-09 | 代码模块化：LLM 接入、TTS、THG 为独立模块，接口清晰 |
| NFR-10 | 配置外部化：LLM URL/Token、音色路径、形象路径等通过环境变量或配置文件注入 |
| NFR-11 | 日志：关键阶段（LLM 首 token、TTS 首包、THG 首帧）记录时间戳，便于延迟排查 |

## 5. 技术约束（不可变更）

### 5.1 硬件约束

- **NPU**：单卡昇腾 910B（容器映射 `/dev/davinci7`，即当前只能使用这一张卡）
- **容器**：基于 `ascend-pytorch:24.0.0-A2-2.1.0-ubuntu20.04` 或同系列 CANN 兼容镜像
- **宿主机**：不可修改，所有操作限制在容器内

### 5.2 软件约束

- **CANN 版本**：必须与宿主机驱动/固件匹配（当前已知宿主机为 CANN 8.0 级别）
- **PyTorch**：`torch_npu` 兼容版本，推荐 PyTorch 2.1.0 + torch_npu 2.1.0
- **NumPy**：必须 < 2.0（铁律）
- **Python**：3.10.x

### 5.3 外部依赖

- **LLM API**：已部署，OpenAI 格式，地址默认为 `http://192.168.1.117:1025/v1`（容器内可访问的本地 MindIE 服务）。外部端点 `modelhub.lgdg.cc` 在当前环境不可达。
- **Token**：从环境变量 `LLM_API_KEY` 读取，当前本地 MindIE 端点无需 API Key。

## 7. 接口定义（前后端契约）

> **说明**：PRD 不指定具体技术栈（WebSocket/gRPC/REST/SSE 由开发 Agent 选择），但必须定义以下接口契约。开发 Agent 需在 `docs/api.md` 中详细说明所选技术栈的实现细节。

### 7.1 对话发起接口

- **输入**：用户文本（string）、音色 ID（string）、语种（string，可选）
- **输出**：流式事件序列（见 7.2）
- **触发方式**：前端用户提交后，后端开始整个对话流程

### 7.2 流式事件序列（后端 → 前端）

后端必须向前端发送以下类型的事件流：

```
[事件 1] 对话开始标记
[事件 2] LLM 文本流（逐字/逐句，可选，用于前端字幕展示）
[事件 3] 首帧视频标记（首帧视频数据到达）
[事件 4] 视频帧流（连续的音视频数据，或视频帧序列）
[事件 5] 对话结束标记
[事件 6] 错误标记（如发生错误）
```

每个事件必须包含：
- `event_type`：事件类型标识
- `timestamp`：事件产生时间戳（毫秒级）
- `payload`：事件载荷

### 7.3 前端 → 后端控制消息

| 消息类型 | 说明 |
|---------|------|
| `start_conversation` | 开始新一轮对话（携带文本、音色 ID） |
| `stop_conversation` | 用户主动终止当前对话 |
| `switch_voice` | 切换音色（作用于下一轮对话） |
| `switch_language` | 切换语言模式（auto/zh/en/dialect） |

### 7.4 错误码定义

| 错误码 | 含义 | 前端处理建议 |
|--------|------|------------|
| `E001` | LLM API 不可用 | 提示用户检查网络或 LLM 服务状态 |
| `E002` | TTS 合成失败 | 降级为静音输出或提示重试 |
| `E003` | THG 渲染失败 | 降级为静态形象 + 音频播放 |
| `E004` | NPU 显存不足 | 提示用户降低并发或重启服务 |
| `E005` | 形象/音色资源缺失 | 提示用户检查资源文件 |
| `E999` | 未知错误 | 记录日志，提示用户重试 |

## 8. Loop 开发阶段定义

本项目采用 **Agentic Loop 方法论** 分阶段实现。每个 Phase 必须有：**明确目标**、**交付物**、**验收标准**、**停止条件**。

开发 Agent 每完成一个 Phase，必须更新 `loop/STATE.md`（格式见 8.2），通过验收后才能进入下一个 Phase。

### 8.1 Phase 总览

| Phase | 名称 | 目标 | 交付物 | 验收标准 | 停止条件 |
|-------|------|------|--------|---------|---------|
| **Phase 0** | 环境基线验证 | 确认容器内 NPU、PyTorch、NumPy 基线可用 | 环境验证报告 | NPU 可见；torch_npu 可用；NumPy < 2.0 | 重试 3 次失败则人工接管 |
| **Phase 1** | 技术选型与架构设计 | 确定技术栈（TTS/THG/通信协议），输出架构方案 | 技术选型文档 `docs/architecture.md` | 方案覆盖所有 Must-Have 功能；有明确的风险评估 | 方案被评审不通过则修订 |
| **Phase 2** | LLM 接入层 | 实现流式 LLM 客户端 | `src/llm_client.py`（或对应模块） | 能流式接收文本；延迟 < 500ms；支持多轮对话 | 重试 3 次失败则人工接管 |
| **Phase 3** | TTS 引擎集成 | 实现 TTS 模块，支持中英文 | `src/tts_engine.py`（或对应模块） | 中文/英文文本 → 语音；音质可接受；支持流式 chunk | 重试 3 次失败则人工接管 |
| **Phase 4** | THG 唇形同步引擎 | 实现 2D 唇形同步渲染 | `src/thg_engine.py`（或对应模块） | 音频 → 唇形同步视频帧；肉眼对齐；≥ 25 fps | 重试 3 次失败则人工接管 |
| **Phase 5** | 实时流式管线串联 | 将 LLM → TTS → THG 串联为流式 Pipeline | `src/pipeline.py`（或对应模块） | 端到端流式运行；首帧延迟 ≤ 1.5s；对话总延迟 < 5s | 重试 3 次失败则人工接管 |
| **Phase 6** | Gradio 前端与接口 | 实现前端页面和接口文档 | `src/webui.py` + `docs/api.md` | Gradio 页面可交互；接口文档完整；前后端通信正常 | 重试 3 次失败则人工接管 |
| **Phase 7** | 集成验收与测试 | 完整系统测试，输出延迟报告 | `docs/benchmark.md` + 验收结论 | 3 组测试通过；中英文对话正常；延迟达标 | 质量不达标则人工接管 |
| **Phase 8** | 视频上传 + PaddleSpeech TTS + 视频生成工作流 | 在保留实时对话能力的基础上，新增用户上传视频并输入文本生成 MP4 的能力 | `src/avatar_manager.py` + `src/video_gen_pipeline.py` + 前端视频生成 Tab | 默认视频上传成功；PaddleSpeech TTS 生成音频；MuseTalk 渲染出 512×512 H.264+AAC MP4 | 重试 3 次失败则人工接管 |

### 8.2 状态持久化机制（Loop 记忆系统）

开发 Agent 必须在 `ascend-avatar/loop/` 目录维护以下状态文件：

```
/ascend-avatar/loop/
├── STATE.md          # 实时状态：当前 Phase、上次心跳、阻塞原因
├── checkpoints/      # Phase 完成标记
│   ├── phase_0.ok
│   ├── phase_1.ok
│   └── ...
├── logs/             # 各 Phase 执行日志
│   ├── phase_0.log
│   └── ...
└── pitfalls/         # 踩坑记录
    ├── phase_1.md
    └── ...
```

**STATE.md 模板**：

```markdown
# ascend-avatar Loop 状态

项目: ascend-avatar
当前 Phase: [Phase 编号]
上次心跳: [YYYY-MM-DD HH:MM:SS]
上次更新者: [Agent 标识]
状态: [RUNNING / DONE / BLOCKED]

## 已完成 Phase
- [x] Phase 0: 环境基线验证
- [ ] Phase 1: 技术选型与架构设计
- ...

## 当前阻塞
[无 / 具体阻塞原因]

## 已知坑
[列出 pitfalls/ 目录中的已知问题]
```

### 8.3 假死检测

开发 Agent 在执行长任务（如模型下载、编译）时，每 5 分钟必须更新 `STATE.md` 的心跳时间戳。如果 30 分钟无心跳，视为假死，人工介入检查。

### 8.4 回退策略（Course Correct）

每个 Phase 验证失败时：
1. 记录失败原因到 `loop/pitfalls/phase_X.md`
2. 自动回退到当前 Phase 起点重新执行
3. 同一 Phase 连续失败 3 次 → 标记为 BLOCKED，等待人工接管

### 7.5 视频生成接口

除实时对话 SSE 外，系统还提供 REST 接口用于“上传视频 → 输入文本 → 生成 MP4”：

| 端点 | 方法 | 说明 |
|------|------|------|
| `/api/upload` | POST | 上传正面人脸视频，返回 `upload_id` |
| `/api/upload/status/{upload_id}` | GET | 查询视频预处理状态 |
| `/api/generate` | POST | 提交文本与 `upload_id`，生成唇形同步 MP4 |
| `/api/generate/status/{job_id}` | GET | 查询生成任务状态 |
| `/api/download/{job_id}` | GET | 下载生成的 MP4 |
| `/api/voices` | GET | 获取 edge-tts 与 PaddleSpeech 可用音色 |

详见 [`docs/api.md`](docs/api.md)。

## 9. 素材与资源要求

### 9.1 默认数字人形象

| 属性 | 要求 |
|------|------|
| 数量 | 1 张 |
| 格式 | JPG / PNG |
| 分辨率 | ≥ 512×512，推荐 1024×1024 |
| 姿态 | 正面或微侧面，胸部以上半身像 |
| 背景 | 纯色或透明（便于后期处理） |
| 表情 | 自然微笑或中性，闭嘴（便于唇形驱动） |
| 服装 | 避免与肤色相近；有领上衣最佳 |
| 备注 | 若无法提供真实素材，可用 AI 生成人像，但需避免版权问题 |

### 9.2 内置音色

| 属性 | 要求 |
|------|------|
| 数量 | ≥ 2 个（建议 1 男 1 女，或 2 种不同风格） |
| 格式 | WAV，16bit 或 32bit，单声道或立体声 |
| 采样率 | 16kHz 或 44.1kHz |
| 时长 | 3~10 秒（零样本克隆参考音频） |
| 内容 | 清晰中文朗读，无背景噪声 |
| 备注 | 若 TTS 引擎使用自研音色库，可不用参考音频文件 |

## 10. 验收标准

### 10.1 功能验收（必须全部通过）

- [x] 在浏览器中打开前端页面，可正常加载无报错

## 12. 附录：关键术语表

| 术语 | 说明 |
|------|------|
| **TTS** | Text-to-Speech，文本转语音 |

| **LLM** | Large Language Model，大语言模型 |
| **NPU** | Neural Processing Unit，昇腾 910B 的神经网络处理器 |
| **DVPP** | Digital Vision Pre-Processing，昇腾数字视觉预处理单元 |
| **CANN** | Compute Architecture for Neural Networks，昇腾神经网络计算架构 |
| **vNPU** | 虚拟 NPU，昇腾 910B 的显存切片技术 |
| **HBM** | High Bandwidth Memory，高带宽显存 |
| **OM** | Offline Model，昇腾离线模型格式（`.om`） |
| **ATC** | Ascend Tensor Compiler，昇腾模型转换工具 |
| **WebRTC** | Web Real-Time Communication，浏览器实时通信协议 |
| **RTMP** | Real-Time Messaging Protocol，实时消息传输协议 |
| **HLS** | HTTP Live Streaming，HTTP 实时流协议 |
| **SDP** | Session Description Protocol，会话描述协议 |
| **ICE** | Interactive Connectivity Establishment，交互式连接建立 |
